# US-069 Deep Audit - Symphony Web UI Controller

## Metadata

- Date: 2026-07-07
- Scope: Symphony Web UI Controller production-readiness audit
- Mode: Review-only audit using `frontend-agents:frontend-production-workflow`
- Primary story: `US-069 Web UI Design Principles And Validation`
- Product contract: `docs/product/symphony-web-ui-controller.md`
- Frontend surface: `crates/harness-symphony/web-ui`
- Harness records: Intake `#169`, Trace `#191`

## Decision

Fix Required.

The controller builds and passes its existing automated checks, but the audit
found unresolved major issues in live board refresh, recovery log state,
review error handling, artifact access, and accessibility. The current state is
good enough for continued iteration, but not clean enough to accept as
production-ready under the frontend production workflow gate.

## UI Contract Reviewed

The product contract expects the Web UI to:

- Act as a local browser controller over Harness/Symphony concepts, not as a
  second source of truth.
- Show all task states: Ready, Blocked, In Progress, Review, Needs Attention,
  and Done.
- Start Ready work, stream run events, show review artifacts, support PR
  merged gating, approve sync, and transition to Done.
- Explain Needs Attention with concrete failure reasons and evidence paths.
- Preserve failed-run evidence during recovery.
- Keep dense operational UI scannable and bounded.
- Keep raw artifacts accessible from the review surface.
- Provide build, browser, desktop/mobile, and design-validation proof.

## Findings

### Major - Board State Does Not Auto-Refresh During Active Runs

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:184`
- `crates/harness-symphony/web-ui/src/main.tsx:413`
- `crates/harness-symphony/web-ui/src/main.tsx:214`

The board loads on mount and can be manually refreshed, but there is no
automatic refresh tied to an active run. A task can complete or fail in the
backend while the UI continues to show it as `In Progress` until the user
presses Refresh.

Impact:

- Breaks the expected workflow from In Progress to Review or Needs Attention.
- Makes the controller less trustworthy as a live operational surface.

Required fix:

- Poll `/api/board` while an active run exists, or refresh the board when the
  active event stream reaches a terminal state.
- Add Playwright tests for `In Progress -> Review` and `In Progress -> Needs
  Attention` without manual refresh.

### Major - Recovered Active Runs Can Show Stale Failed-Run Communication

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:923`
- `crates/harness-symphony/web-ui/src/main.tsx:1087`
- `crates/harness-symphony/web-ui/tests/board.spec.ts:490`

The detail view preserves old review state during execution recovery, which is
useful for failed-run evidence. However, the event log prefers
`review?.events` over live polled `events`. After retrying a failed task, the
UI can show the new run as active while still rendering old failed-run
communication.

Impact:

- Confuses recovery review.
- Makes it hard to tell whether the current active retry is actually
  progressing.

Required fix:

- Separate failed-run evidence from active-run communication.
- When `item.active_run` exists, show live active-run events in the main run
  communication area.
- Keep prior failed-run artifacts in a distinct evidence panel.
- Add E2E coverage that checks the retry shows new-run events while old
  evidence remains available.

### Major - Review Endpoint Failures Are Silent In The UI

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:931`
- `crates/harness-symphony/web-ui/src/main.tsx:1073`

The review loader only updates state when the response is OK. If
`/api/runs/<run-id>/review` returns 404, 500, or malformed data, the review
panel simply does not render. The user sees no explicit failure, retry state,
or artifact explanation.

Impact:

- Violates the Needs Attention and review-surface contract.
- Hides the exact place where technical maintainers need evidence.

Required fix:

- Add explicit review loading/error state.
- Render a review-error panel with status code and suggested action.
- Add E2E coverage for a failing review endpoint.

### Major - `Open artifacts` Is An Enabled No-Op

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:1048`
- `docs/product/symphony-web-ui-controller.md` review surface contract

The detail controls render an enabled `Open artifacts` button without
`onClick`, link target, disabled state, or explanation.

Impact:

- User-facing dead control.
- Contradicts the requirement that raw artifacts remain accessible from the
  review surface.

Required fix:

- Wire the control to artifact paths when artifacts exist, or remove it.
- If artifact opening is not implemented yet, render it disabled with a clear
  reason.
- Add E2E coverage that the artifact control either works or is explicitly
  unavailable.

### Major - Custom Dialog Declares Modal Semantics Without Modal Behavior

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:775`
- `crates/harness-symphony/web-ui/src/main.tsx:882`
- `crates/harness-symphony/web-ui/src/main.tsx:988`

The task detail popup declares `aria-modal="true"` and focuses the dialog
container on open. It does not implement focus trapping, inert background,
proper focus restoration to the opened task card, or targeted keyboard tests.

Impact:

- Keyboard users can tab into background content while a modal is open.
- Screen-reader semantics overstate actual modal behavior.
- Closing the dialog can leave focus in an unpredictable place.

Required fix:

- Use a real Dialog primitive or implement equivalent behavior:
  focus trap, inert background, labelled title, Escape handling scoped to the
  dialog, and focus restoration.
- Add Playwright tests for tab containment, Escape close, backdrop behavior,
  and focus restoration.

### Major - Search Input Has No Durable Accessible Name

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:404`

The search input is inside an empty `<label>` and relies on the placeholder
`Find task`. Placeholder text is not a durable accessible label.

Impact:

- Assistive technology may not expose a reliable control name.
- The search control is central to scanning the board.

Required fix:

- Add visible label text or `aria-label="Find task"`.
- Prefer visible or screen-reader-only label text over placeholder-only
  labeling.
- Add an accessibility-oriented Playwright assertion for the search field name.

### Major - Review/Action Loading And Error States Lack Status Semantics

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:156`
- `crates/harness-symphony/web-ui/src/main.tsx:422`
- `crates/harness-symphony/web-ui/src/main.tsx:429`
- `crates/harness-symphony/web-ui/src/main.tsx:1223`

The UI has visual loading/error cues, but the board does not expose `aria-busy`,
the error card has no `role="alert"`, and run/event updates are not exposed as
live status updates.

Impact:

- Screen-reader users may not know when loading starts, fails, or updates.
- Initial board load can look like empty columns before data is known.

Required fix:

- Add `aria-busy` for board loading.
- Use `role="alert"` for action failures.
- Use `role="status"` or an appropriate live region for non-error progress
  updates.
- Add tests for alert/status semantics.

### Advisory - `main.tsx` Is Now The Feature Boundary

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx` is 1,306 lines.
- It contains API DTOs, fetch actions, board rendering, sidebar, detail dialog,
  review panel, event log, confetti, local hooks, and page composition.

Impact:

- The feature is harder to review and evolve safely.
- Small changes can cut across unrelated concerns.
- Repeated product-specific patterns are not yet extracted despite the product
  contract calling for extraction when patterns repeat.

Recommended fix:

- Extract feature-owned modules for:
  - API/types.
  - Board grid and card.
  - Sidebar/dependency graph.
  - Task detail dialog.
  - Review panel.
  - Event polling and run communication.
  - Controller hooks/actions.
- Keep `components/ui/*` generic.

### Advisory - Fetch Concurrency Is Not Ordered Or Abortable

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:167`
- `crates/harness-symphony/web-ui/src/main.tsx:886`
- `crates/harness-symphony/web-ui/src/main.tsx:920`

`loadBoard` commits whichever request resolves last. Manual refresh and
action-triggered refreshes can overlap without an abort controller or
monotonic request guard. Event/review effects use cancellation flags for
unmount safety but do not abort in-flight requests or normalize stale
response behavior.

Impact:

- Older responses can overwrite newer board state in edge cases.
- Review and event behavior can be hard to reason about under rapid user
  actions.

Recommended fix:

- Add request ordering guards or `AbortController` for board/review/event
  requests.
- Add tests where an older board response resolves after a newer one.

### Advisory - Historical Event Polling Is Too Broad

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:886`

The detail view polls `/api/runs/<run-id>/events` every two seconds whenever
there is `item.active_run ?? item.run_id`, including Review, Done, or other
non-active historical runs. Review payloads already include events.

Impact:

- Unnecessary backend polling for historical detail views.
- Makes future performance behavior noisier as run count and log size grow.

Recommended fix:

- Poll only active runs.
- Use review payload events for historical review/done states.
- Stop or back off polling after terminal events.

### Advisory - US-069 Design Validation Remains Planned

Evidence:

- `docs/stories/epics/E08-symphony-web-ui-controller/US-069-web-ui-design-principles-validation.md`
- `crates/harness-symphony/web-ui/playwright.config.ts`
- `crates/harness-symphony/web-ui/tests/board.spec.ts`

US-069 is still planned and has no evidence recorded. The current Playwright
suite validates behavior and overflow but does not capture screenshot proof,
visual regression, console/network health, or design-tool skip evidence. It
also checks desktop `1440x820` and mobile `390x760`, but not tablet or
split-screen widths.

Impact:

- The product contract says design validation should combine deterministic
  proof with visual review.
- Intermediate responsive density can regress without test coverage.

Recommended fix:

- Complete US-069.
- Add screenshot or equivalent visual proof.
- Add an intermediate viewport, such as tablet/split-screen.
- Document clean skip when Impeccable or equivalent tooling is not registered
  or installed.

### Advisory - Intermediate Widths May Force Unreviewed Horizontal Board Scrolling

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:688`
- `crates/harness-symphony/web-ui/tests/board.spec.ts:285`

The board uses a `min-w-[1120px]` six-column grid until `max-sm`, inside an
overflow-x container. That may be acceptable for a dense controller, but it is
not explicitly validated at tablet or split-screen widths.

Impact:

- A common viewport class may become horizontally scroll-heavy.
- Users on laptop split view or tablet widths may get a worse board experience
  than desktop/mobile tests suggest.

Recommended fix:

- Add responsive assertions at an intermediate width.
- Decide whether the board should reflow before `max-sm` or intentionally
  remain horizontally scrollable with affordances.

### Advisory - Long Detail And Review Values Can Overflow Or Degrade

Evidence:

- `crates/harness-symphony/web-ui/src/main.tsx:1283`
- `crates/harness-symphony/web-ui/src/components/ui/badge.tsx:24`

`ListBlock` renders arbitrary strings as compact badges. Long artifact paths,
IDs, or changed file paths may wrap poorly or overflow in the detail popup.
Board cards are bounded, but detail/review values need their own constraints.

Recommended fix:

- Use wrapping rows or max-width detail chips for long values.
- Add detail-popup overflow assertions for long artifact paths and changed
  files.

### Advisory - Local Shadcn-Style Layer Is Too Thin For Current Complexity

Evidence:

- `crates/harness-symphony/web-ui/package.json`
- `crates/harness-symphony/web-ui/src/components/ui/button.tsx`
- `crates/harness-symphony/web-ui/src/components/ui/card.tsx`

The app has local shadcn-style primitives, but no Dialog, Alert, Skeleton,
Tooltip, or Radix primitive dependency. Modal, alert, and loading behavior are
hand-rolled in page code.

Impact:

- Accessibility and state gaps are more likely.
- UI behavior depends on custom implementation instead of well-tested
  primitives.

Recommended fix:

- Add or implement proper primitives for Dialog, Alert/Status, Skeleton, and
  Tooltip where behavior warrants them.
- Keep primitives generic and feature logic outside `components/ui`.

### Low - Status Colors Partly Bypass Tokens

Evidence:

- `crates/harness-symphony/web-ui/src/components/ui/badge.tsx:6`
- `crates/harness-symphony/web-ui/src/styles.css:5`

`Badge` hardcodes emerald, zinc, blue, violet, red, and teal classes instead
of mapping through semantic status tokens.

Impact:

- Not currently blocking.
- Makes state contrast and theme consistency harder to audit later.

Recommended fix:

- Add semantic status tokens and map badges to those tokens.
- Include contrast checks when visual validation tooling exists.

### Low - Build-Only Vite Plugin Is In Runtime Dependencies

Evidence:

- `crates/harness-symphony/web-ui/package.json:18`

`@vitejs/plugin-react` is in `dependencies` rather than `devDependencies`.

Impact:

- Low current risk.
- Can unnecessarily ship build-only packages into Electron packaging context
  or confuse dependency ownership.

Recommended fix:

- Move `@vitejs/plugin-react` to `devDependencies`.

## Existing Strengths

- Build, backend web tests, browser E2E, and desktop smoke all pass.
- The local API contract is covered by Rust web tests.
- Needs Attention failure summaries and recovery controls are represented in
  existing Playwright coverage.
- Board card overflow has explicit desktop and mobile E2E assertions.
- Ready-only delete behavior is guarded in UI and backend tests.
- `components/ui/*` remains generic; product-specific leakage is mostly in the
  page/feature layer.

## Recommended Fix Order

1. Fix live state correctness:
   - Active-run board refresh.
   - Active retry event log versus failed-run evidence separation.
   - Review endpoint error state.

2. Fix dead or misleading controls:
   - `Open artifacts`.
   - Review loading/error affordances.

3. Fix modal and search accessibility:
   - Dialog focus trap/restoration/inert behavior.
   - Search accessible name.
   - Alert/status semantics.

4. Add missing tests:
   - Active transition tests.
   - Recovery live event test.
   - Review fetch failure test.
   - Keyboard/focus tests.
   - Artifact control behavior.
   - Intermediate responsive viewport.

5. Complete US-069:
   - Lightweight design contract.
   - Screenshot or equivalent visual proof.
   - Design-tool clean skip if unavailable.

6. Refactor for maintainability:
   - Split `main.tsx` into feature modules.
   - Add missing UI primitives where warranted.
   - Normalize semantic status tokens.

## Subagent Decisions

Architecture and React quality:

- Decision: Fix Required.
- Key findings: no active board auto-refresh, stale events after recovery,
  oversized `main.tsx`, unordered fetch concurrency.

Shadcn-style composition and accessibility:

- Decision: Fix Required.
- Key findings: custom modal lacks focus containment/restoration, search has no
  durable accessible name, loading/error states lack status semantics, artifact
  button is dead.

Vite/performance and design polish:

- Decision: Fix Required.
- Key findings: historical event polling is too broad, US-069 visual
  validation remains open, intermediate responsive density is untested, long
  detail values need bounding, `@vitejs/plugin-react` belongs in
  devDependencies.
